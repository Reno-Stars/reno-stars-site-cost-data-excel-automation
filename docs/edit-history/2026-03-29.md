# 2026-03-29 — Payment Rows Available for Worker Data + Code Review Fixes

## Summary

Iterative fix to eliminate gaps in worker data placement. The core principle established: each column group (payment A/B, project C, date D, worker D/E/F/G/H/I, material K/L/M) is independent. A row is available for worker data if D and E are both empty, regardless of other column values. Followed by two rounds of comprehensive code review fixing all critical, warning, and suggestion issues — covering named constants, security hardening, function extraction, accessibility, performance, and documentation alignment.

## Bug Fixes

### Row availability check too restrictive (user-reported, 3 iterations)

**Symptom:** Gaps between worker entries — rows with material data but no worker name were skipped; payment rows were skipped entirely.

**Iteration 1 — `isRowEmpty` checked too many columns:**
- Original function checked C/D/E/F/G/K — all had to be empty
- Fix: Renamed to `isRowAvailableForWorker`, changed to check only E (worker name)
- Result: Still gaps — payment rows still skipped by `findOrCreateInsertRow`

**Iteration 2 — Added D check (wrong approach):**
- Changed to check D AND E both empty
- This was wrong — sent workers to buffer rows instead of filling payment rows

**Iteration 3 — Removed payment row skip (correct fix):**
- User clarified: "row 168 169 170 should be filled" — payment rows (第一笔款/第二笔款/第三笔款) should receive worker data
- Payment info (A/B) and worker data (D/E/F/G/H/I) are independent
- Removed `PAYMENT_LABELS` skip from `findOrCreateInsertRow`
- Now any row where D and E are both empty is available, including payment rows

**Key principle:** Each column group is independent:
- Payment info: A, B
- Project name: C
- Worker data: D, E, F, G, H, I
- Material data: K, L, M
